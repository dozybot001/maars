Research

Research is the core stage of MAARS. The Python runtime orchestrates task decomposition, parallel execution, verification, and iterative evaluation. Each atomic task runs inside a Docker sandbox under agent control.

01How it fits in

Research sits between Refine (which sharpens the research idea) and Write (which drafts the paper). It receives a refined_idea.md and produces a set of task results, artifacts, and an evaluation history that Write later synthesises into prose. The stage may run multiple outer iterations — each iteration refreshes Strategy, re-decomposes the plan, re-executes tasks, and evaluates completeness.

02Preflight

_preflight_docker() is the first call inside _execute(). It verifies three conditions before spending any tokens:

  • Docker SDK installed (pip install docker)
  • Docker daemon reachable (client.ping())
  • Sandbox image built — MAARS_DOCKER_SANDBOX_IMAGE (default maars-sandbox:latest)

Any failure raises immediately — no wasted calibration tokens.

03Principles

  • Every LLM call receives a capability profile (_build_capability_profile) — sandbox config + available tool list — so the model knows what it can and cannot do.
  • Context is explicit: Calibrate → Strategy → Decompose → Execute → Evaluate. Each phase passes its output forward; nothing is re-derived implicitly.
  • plan_tree.json is the single source of truth. plan_list.json is a derived cache — flat view of the tree for convenience.

Calibrate runs once per session. Strategy through Evaluate repeat every outer iteration.

05Tool scoping

Each phase receives only the tools it actually needs. Planning phases cannot run code; evaluation does not search for new papers.

Phase Tools Rationale
Calibrate none Pure reasoning from the capability profile text; no tools needed.
Strategy arXiv · Wikipedia Search for best practices only. No code_execute — must search, not run experiments.
Decompose root read + search May inspect existing results and search for domain context; must not execute code during planning.
Decompose non-root none JSON output only.
Execute full set Handles both code tasks (code_execute) and literature tasks (search), plus result inspection via read tools.
Verify none JSON verdict only.
Evaluate read tools Inspect completed results only; no code_execute, no search — must not run new experiments or fetch new papers.

Read tools: read_task_output · list_tasks · read_refined_idea · read_plan_tree · read_results_summary · read_artifact_file · list_artifacts. The capability profile passed to Calibrate / Strategy / Evaluate always lists the full Execute tool set, so planning agents understand what execution tasks can do — even though they themselves cannot call those tools.

06Main loop skeleton

Calibrate once, then iterate: Strategy → Decompose → Execute → Evaluate, until Evaluate emits no strategy_update.

async def _execute(self):
    await _preflight_docker()             # abort early if docker not ready

    await _calibrate_once(idea)          # run once per session

    evaluation = None
    while True:
        await Strategy(idea, evaluation)   # None on first round
        await Decompose(idea, strategy)
        await Execute(tasks)               # parallel, includes verify/retry/redecompose
        evaluation = await Evaluate(results, score)
        if not evaluation.strategy_update: break
        iteration += 1

Evaluate uses a prefer-stop policy: it only emits strategy_update if there is a critical gap that the current plan genuinely cannot cover. Minor issues or diminishing-returns directions do not trigger another iteration. When is_final=True is set externally, Evaluate produces a summary and does not emit strategy_update.

08Parallel execution model

Tasks respect dependency ordering via topological batching; concurrency is capped by semaphore.

# topological batching — respects dependency DAG
batches = topological_batches(tasks)

for batch in batches:
    results = await asyncio.gather(
        *[execute_task(t) for t in pending],
        return_exceptions=True,            # one task failure won't cancel siblings
    )
Copied ✓

Each execute_task acquires _get_api_semaphore() before calling _stream_llm, bounding concurrent LLM calls to MAARS_API_CONCURRENCY. The Execute → Verify → (pass | retry | redecompose) cycle is atomic inside the semaphore.

Rate-limit config

  • MAARS_API_CONCURRENCY — maximum number of LLM calls in flight simultaneously.
  • MAARS_API_REQUEST_INTERVAL — minimum seconds between consecutive LLM calls (enforced inside _stream_llm via _rate_limit()).

09_stream_llm internals

Single entry-point for all LLM calls, in stage.py.

  • Rate limiting: calls _rate_limit() before every request, enforcing the MAARS_API_REQUEST_INTERVAL gap.
  • Model isolation: deep-copies the model per Agent instance (deepcopy(model)). Prevents shared-state bugs when multiple tasks call _stream_llm concurrently.
  • No validate parameter: JSON structure validation belongs to the call site, via for/else retry loops — not inside _stream_llm.

10JSON parsing + retry

parse_json_fenced

Extracts JSON from an LLM-generated fenced code block. Applies _repair_json_escapes() before parsing — fixes common LaTeX backslash sequences (e.g. \rho, \lambda) that would otherwise break json.loads.

Call-site retry loop

Decompose judge, Reviewer, Verify, and Evaluate all use a for/else retry loop (2 attempts). On first parse failure the LLM is called again; if both attempts fail, a safe fallback value is used. This replaces the previous pattern of inline fallback without retry. 

for attempt in range(2):
    raw = await _stream_llm(agent, prompt)
    try:
        result = parse_json_fenced(raw)
        break
    except ValueError:
        continue
else:
    result = SAFE_FALLBACK    # both attempts failed
